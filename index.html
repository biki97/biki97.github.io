AI / ML 5 MIN READ · By Biki Dutta

Why model accuracy isn't enough — the case for macro recall in food-tech vision

A few months into my work at Borde, I learned a humbling lesson: accuracy is the easiest metric to fool yourself with. If 95% of the dataset is one class, a model that predicts that class for every image will score 95% accuracy — and be completely useless in production.

In food-tech computer vision, the stakes are real. A walnut sorting model that misses 80% of Red_Pieces (a rare defect class) still looks great on a confusion matrix dominated by FM_Shell and Light_Pieces. The rare class disappears in the average.

Macro recall vs micro recall

The fix isn't complicated, but it took me a while to internalize it:

  • Micro recall — totals true positives and false negatives across all classes. Dominated by majority classes.
  • Macro recall — computes recall per class, then averages. Every class is treated equally — including the rare ones.

When you actually care about catching every defect type, macro recall is the honest metric. A 95% accuracy model with 30% macro recall is a model that's failing where it matters most.

What I do now

Every time we evaluate a new training run, I look at three things before celebrating: the confusion matrix (where are the off-diagonal errors clustering?), macro recall (are we catching the rare classes?), and false positives per class (where is the model crying wolf?).

The model that "looks worse" on overall accuracy but has cleaner per-class recall is almost always the better model to ship. Accuracy is a vanity metric. Recall — broken down by class — is the truth.

Engineering 4 MIN READ · By Biki Dutta

From Spring Boot to Roboflow — what backend engineering taught me about ML data pipelines

I came to ML data work from a backend internship, where I was building REST APIs in Spring Boot on top of PostgreSQL. On the surface, that's nothing like training a CNN on Roboflow. But the deeper I get into ML workflows, the more I notice how much they look like backend systems with extra steps.

Three patterns that carry over

1. Validation at the edge. In Spring Boot, you don't trust user input — you validate it before it ever touches business logic. In ML, you don't trust training data — you check class balance, annotation quality, and edge cases before pressing "train." Same instinct, different layer.

2. Idempotent rebuilds. Good backend systems can rebuild from source. Good ML systems can re-run from the same dataset and produce the same model. Versioning your data (and your data is the actual source) is just git for machine learning.

3. Observability over hope. When a backend endpoint fails, you have logs, traces, and metrics. When an ML model fails, you have… vibes? Not anymore. Confusion matrices, per-class recall, false-positive breakdowns — these are the production telemetry of an ML system.

Where it gets interesting

The thing backend engineering didn't prepare me for is the fuzziness. A REST endpoint either returns 200 or it doesn't. A model can be "kind of working" — high accuracy, terrible recall on the rare class that actually matters. There's no green build status for "it generalizes well to data we haven't seen yet."

That's the hard, interesting part. And weirdly, the backend engineer in me is finding the ML world's lack of standard observability patterns kind of exciting. There's a lot of room to bring engineering discipline to a field that's still maturing.
