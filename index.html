Why model accuracy isn't enough — the case for macro recall in food-tech vision

A few months into my work at Borde, I learned a humbling lesson: accuracy is the easiest metric to fool yourself with. If 95% of the dataset is one class, a model that predicts that class for every image will score 95% accuracy — and be completely useless in production.

In food-tech computer vision, the stakes are real. A walnut sorting model that misses 80% of Red_Pieces still looks great on a confusion matrix dominated by FM_Shell. The rare class disappears in the average.

Macro recall vs micro recall

  • Micro recall — totals true positives and false negatives across all classes. Dominated by majority classes.
  • Macro recall — computes recall per class, then averages. Every class is treated equally.

When you actually care about catching every defect type, macro recall is the honest metric. A 95% accuracy model with 30% macro recall is a model that's failing where it matters most.

What I do now

Every time we evaluate a new training run, I look at three things: the confusion matrix, macro recall, and false positives per class. The model that "looks worse" on overall accuracy but has cleaner per-class recall is almost always the better model to ship.

From Spring Boot to Label Studio — what backend engineering taught me about ML data pipelines

I came to ML data work from a backend internship building REST APIs in Spring Boot. The deeper I get into ML workflows, the more I notice how much they look like backend systems with extra steps.

Three patterns that carry over

1. Validation at the edge. In Spring Boot, you don't trust user input. In ML, you don't trust training data — you check class balance, annotation quality, and edge cases before training.

2. Idempotent rebuilds. Good ML systems can re-run from the same dataset and produce the same model. Versioning your data is just git for machine learning.

3. Observability over hope. Confusion matrices, per-class recall, false-positive breakdowns — these are the production telemetry of an ML system.

Where it gets interesting

A REST endpoint either returns 200 or it doesn't. A model can be "kind of working." That's the hard, interesting part — and weirdly exciting to bring engineering discipline to.

Building AI Email Writer — what shipping a Gemini-powered React app taught me about AI product design

A few months ago I had a simple frustration: writing professional emails is slow. Every time I needed to send a formal request, follow up, or draft a cold outreach, I'd spend more time on wording than on the actual content. So I built something to fix that — AI Email Writer, a live React app powered by Google's Gemini API.

The project is live at ai-email-writer-jade.vercel.app. Here's what I learned building it.

The stack decision

I chose React + Vite for the frontend — fast dev server, clean component model, and easy to deploy. For the AI layer, I went with Google Gemini API: it handles long-context prompts well, which matters when you're giving the model detailed context about tone, recipient, and purpose. Deployment on Vercel meant the feedback loop from code change to live URL was about 30 seconds.

The hard part: prompt engineering

The UI is simple. The hard work is invisible — in the system prompt. My first version just said "write an email about this." Outputs were fine but generic. Real improvement came when I treated the prompt like a structured brief:

  • Context — what the email is about, who it's for
  • Tone — formal, friendly, assertive, concise
  • Goal — request, follow-up, introduction, apology
  • Length constraints — short and punchy vs. detailed

Giving the model all four parameters in a well-structured prompt brought output quality from "usable" to "I would actually send this."

UX decisions that mattered

The first version had no copy button. Users had to manually select and copy the output — a tiny friction that killed the experience. Adding a one-click copy to clipboard button and a subtle generating state animation made the whole thing feel polished. Small details matter when the core loop is this tight.

What I took away

Building this changed how I think about AI tools. The model is a detail — the interface is the product. How you collect context, structure the prompt, and present the output are all design decisions that completely change the user experience. A thoughtful wrapper around a good model produces something that feels like magic. That's what I'm chasing with everything I build next.
